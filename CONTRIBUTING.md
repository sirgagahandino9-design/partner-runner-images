## How to Contribute

### 1. Report Issues

If you find a bug or have a suggestion for the Arm-based images:

1. Go to [actions/runner-images/issues](https://github.com/actions/runner-images/issues)
2. Search to see if your issue already exists
3. If not, create a new issue with:
   - A clear title and description
   - Steps to reproduce (for bugs)
   - The specific image affected (`ubuntu-24.04-arm`, `ubuntu-22.04-arm`, or `windows-11-arm`)
   - Your GitHub Actions workflow (if relevant)

### 2. Request Features or Tools

To request new tools or software on Arm images:

1. Go to [actions/runner-images/issues](https://github.com/actions/runner-images/issues)
2. Create an issue with:
   - What tool or software you need
   - Why it's important for your workflows
   - Which image(s) you need it on
   - Any relevant links or documentation

### 3. Propose Changes

For code or documentation changes:

1. Check existing issues and pull requests first
2. Create an issue describing your proposed change
3. Wait for feedback from maintainers
4. If approved, submit a pull request with:
   - Clear description of changes
   - Link to related issues
   - Any relevant test results or validation

### 4. Documentation

Help improve documentation by:

1. Fixing typos or unclear sections
2. Adding examples or troubleshooting guides
3. Improving image documentation
4. Updating outdated information

## Contribution Guidelines

### Before Contributing

- Familiarize yourself with the [README.md](README.md)
- Read the [SUPPORT.md](SUPPORT.md) for common questions
- Check existing issues to avoid duplicates
- Review the [SECURITY.md](SECURITY.md) for vulnerability reporting

### Code of Conduct

All contributors are expected to follow GitHub's Community Guidelines:
- Be respectful and inclusive
- Provide constructive feedback
- Report inappropriate behavior to GitHub

### Pull Request Process

1. **Fork the repository** (in actions/runner-images)
2. **Create a feature branch** from `main`
3. **Make your changes** with clear commit messages
4. **Test your changes** thoroughly
5. **Submit a pull request** with:
   - Clear description of what changed and why
   - Reference to related issues
   - Screenshots or examples (if applicable)
6. **Address feedback** from maintainers
7. **Wait for approval** and merge

### Commit Message Guidelines

Use clear, concise commit messages:

```
[Type] Short description (50 chars max)

Longer explanation if needed (wrap at 72 chars).

- Use bullet points for multiple changes
- Reference issues: "Fixes #123"
```

Types: `feat`, `fix`, `docs`, `test`, `chore`, `refactor`

---

## Review Process

When you submit a pull request:

1. Automated checks run (linting, tests, etc.)
2. Maintainers review your changes
3. You may be asked for revisions
4. Once approved, your changes are merged
5. Your contribution is now part of the project!
